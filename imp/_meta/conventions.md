# Substrate conventions

This is the per-substrate config doc — operational reference for
authors and tools. Canonical design rationale lives in imp's source
repo at `project/substrate-layers.md`; read that first if you're new
to the layered model.

## Layout

```
<repo>/
  plans/, bugs/, TODO.md, rules/   # human-owned, code-adjacent
  imp/                             # gnome-maintained substrate
    learnings/                       layer 1: gnome-distilled rationale
    reference/                       layer 1: archived external sources
    concepts/                        layer 2: narrative views (selective)
    _index/                          layer 2: per-file/symbol/feature views
    note/{inbox,processed,discarded}/  capture surface
    log.md                           append-only chronological history
    _meta/                           this dir
  .imp/                            gitignored layer-0 cache (build artefact)
```

`rules/` is substrate-shaped (frontmatter, drift tracking) but
human-authored, which is why it lives at the repo root next to the
other human-owned dirs.

## Kinds

Every layer-1 entry has a `kind`. Kinds differ in their relationship
to code, drift semantics, and lifecycle.

### `rule` (root `rules/`)
Locked-in project invariants — design constraints, file formats,
API shapes, anything where breaking it is alarming. Truth lives in
the doc; code is expected to obey. Drift between rule and code is
an **alarm**.

Lifecycle: slow, hand-curated. Human-authored directly; gnome flags
drift but doesn't edit the body.

### `learning` (`imp/learnings/`)
Discovered knowledge — why-decisions, post-incident reasoning,
gotchas, history. Not a rule, not a fact about now; an awareness
signal that survives refactor. The "snake in the back yard" framing.

Lifecycle: aging. Old learnings fade in surfaced views but don't go
false. Authored by the gnome from notes; humans review proposals.

### `plan` (root `plans/`)
The primary working document for any coherent piece of work —
features, refactors, investigations. Most new work starts as a plan
in `state: exploring`. Research, options, decisions accumulate in
the plan doc as work firms up. When a plan ships, learnings may be
distilled out.

States (`state:` frontmatter):
- `exploring` — pre-decision; research and shaping
- `active` — committed; in progress
- `shipped` / `shelved` / `abandoned` — concluded; flat folder, not
  a separate `archive/` dir (use frontmatter, not directory split)

Companion dir: `plans/<slug>/` may sit alongside `plans/<slug>.md`
for prototypes/scratch. The `.md` stays canonical; the dir is
"random crap is fine."

### `reference` (`imp/reference/`)
Pointers to external sources — blog posts, papers, docs, anything
that influenced project direction. Each reference includes an
archived snippet (gnome fetches and saves locally; also submits to
Wayback Machine) so it survives link rot.

Authored by the gnome from notes that include URLs; humans review
proposals.

### `concept` (`imp/concepts/`) — special
Layer-2 narrative views, not authored directly. Generated by
`imp tidy` (or `imp wiki`) from layer-1 rationale entries with shared
tags. Don't hand-edit; changes are overwritten on regeneration.

Concept pages are produced selectively, not exhaustively — only
where prose synthesis pays off (contested designs, cross-cutting
concerns). For the cheap default per-file view, see `imp/_index/by-file/`.

## Frontmatter

Every layer-1 entry uses YAML frontmatter, one field per line
(canonical line format — `imp/_index/` ripgrep filters depend on
this):

```yaml
---
kind: rule | learning | plan | reference
title: short human-readable title
created: YYYY-MM-DD
updated: YYYY-MM-DD
status: current | stale | superseded
touches:
  files: [path/to/file.cs, ...]
  symbols: ["scip-symbol-id", ...]
  features: [feature-tag, ...]
provenance:
  author: human | imp-gnome | imp-research:R-NNN | migration:M-NNN
verified-against:
  - { file: path/to/file.cs, hash: <12-char>, lines: 100-150 }
---
```

Per-kind extensions:

| Kind | Extra fields |
|---|---|
| `rule` | `enforces` (optional list of file globs) |
| `learning` | `relevance-horizon` (optional fade date), `topics` (list) |
| `plan` | `state: exploring \| active \| shipped \| shelved \| abandoned`, `supersedes` (list) |
| `reference` | `subject`, `url`, `archived-at` (local snippet path), `wayback-url` |

`touches:` is the join key for layer-3 lookups in `imp/_index/`.

## Drift semantics

Drift = a claim in the substrate disagrees with code or with another
claim. The right response varies by kind:

- **Rule vs code**: **alarm**. Gnome surfaces it; fix the code or
  update the rule deliberately (via proposal — rules are root, human
  territory).
- **Rule vs rule**: **alarm**. Substrate is inconsistent.
- **Learning vs code**: **awareness signal**. The learning may still
  apply, or may have been mooted by a rewrite. `verified-against`
  hash mismatch flips `status: current` → `status: stale`.
- **Plan vs code** (`exploring`/`active`): tracking — drift = "not
  done yet."
- **Plan vs code** (`shipped`/`shelved`/`abandoned`): noise. Suppress.
- **Reference vs source**: **lint**. The local snippet is canonical;
  the upstream URL may have changed.

## Trust model

Imp's writes are partitioned by directory ownership.

**Direct-write (imp's territory — no proposal needed):**
- `imp/learnings/`, `imp/reference/`, `imp/concepts/`, `imp/_index/`
- `imp/log.md`
- `imp/note/{processed,discarded}/`

The gnome commits these directly under a distinct git author
(`imp-gnome <noreply@imp.local>`, set per-commit). Auditability via
`git log --author=imp-gnome` or `git log -- imp/`.

**Proposal-required (cross-boundary):**
- `rules/`, `plans/`, `bugs/`, `TODO.md` (root-level human dirs)

The gnome writes proposals to `<repo>.imp-proposals/P-NNN-<slug>.md`.
The `/imp-promote` skill reviews and applies. Most nights it's empty.

**Out of scope for the gnome:**
- Code edits. Imp's *build mode* handles those in worktrees with its
  own proof-of-work pipeline.

## Auto-approval gradient (proposals only)

When Claude reviews `<repo>.imp-proposals/` on the human's behalf:

- **Always-safe**: `TODO.md` appends.
- **Claude-approvable**: plan edits (append-section), plan
  state-flips, new exploring plans.
- **Human-required**: any change to `rules/`, deletions, anything
  that loses information.

Full tier table in the imp source: `skills/imp-promote.md`.

## Note capture

Notes are pre-classification. Humans/agents drop captures into
`imp/note/inbox/<timestamp>-<slug>.md` via `imp note "<text>"`. The
gnome processes inbox items into structured layer-1 entries on the
next `imp tidy` run; processed source notes move to `processed/`,
discards to `discarded/<reason>/`.

Notes have minimal frontmatter (`captured`, `repo`, `source`,
`git-head`); structuring is the gnome's job.

## Configuration

Optional `_meta/config.yaml`:

```yaml
location: imp/                            # substrate location, relative to repo root
proposals: ../nb.imp-proposals/     # imp's proposal sidecar (gitignored)
```

Defaults are sensible; missing config is fine.
